# Generate disposable local load

Use the load generator to populate dashboards, exercise mixed block types, and
compare engine behavior under a repeatable workload. It intentionally does not
send API-key headers, so it belongs only beside an isolated insecure server and
a disposable database.

**Starting point:** Node.js 20 or newer and the `orch8-server` binary in
`PATH`. Do not point this guide at the Level 3 database.

**Result:** a seeded 30-second workload produces realistic workflows, worker
tasks, signals, and cron activity on port 18080, then the database is removed.

## 1. Create an isolated server project

From the repository root:

```bash
mkdir -p quickstart-work
orch8 init quickstart-work/load-test
cd quickstart-work/load-test
```

This directory and its SQLite database are disposable. Confirm you are not
using the Level 3 path before continuing:

```bash
pwd
test ! -e orch8.db
```

## 2. Start an explicitly insecure local engine

In terminal A, from `quickstart-work/load-test`:

```bash
export ORCH8_HTTP_ADDR=127.0.0.1:18080
export ORCH8_API_KEY=
export ORCH8_REQUIRE_TENANT_HEADER=false
export ORCH8_ALLOW_NO_TENANT_ISOLATION=1
orch8-server --config orch8.toml --insecure
```

The empty API-key override clears the key generated by `orch8 init`;
`--insecure` then acknowledges missing API authentication and unencrypted
storage. The two tenant variables disable and explicitly acknowledge the loss
of tenant isolation because the current load generator sends neither header.
This setup is acceptable only because the process binds to loopback and the
database will be deleted after the exercise.

Check readiness:

```bash
curl -fsS http://127.0.0.1:18080/health/ready | jq
```

## 3. Run a bounded, repeatable workload

In terminal B, from the repository root:

```bash
cd loadgen
npm install
npm start -- \
  --server=http://127.0.0.1:18080 \
  --preset=light \
  --duration=30 \
  --seed=4242 \
  --tenants=loadgen-demo \
  --namespaces=benchmark
```

The fixed seed makes catalog selection and generated inputs repeatable. Timing
still depends on the host and scheduler, so compare distributions rather than
expecting byte-identical metric output.

## 4. Observe while it runs

From a third terminal:

```bash
curl -fsS 'http://127.0.0.1:18080/instances?tenant_id=loadgen-demo&limit=10' |
  jq '.items | map({id, state, sequence_id})'

curl -fsS http://127.0.0.1:18080/metrics |
  grep -E 'orch8_(instances|steps|worker|tick)'
```

The load generator bootstraps a catalog with parallel, routing, loop,
try/catch, race, approval, subsequence, and data-pipeline behavior. LLM traffic
remains off unless explicitly enabled.

## 5. Interpret the final counters

When the 30-second duration ends, the process drains in-flight HTTP and prints
a final snapshot. Record at least:

- submitted and completed instances;
- failed requests and instances;
- observed latency percentiles;
- maximum concurrency; and
- worker task throughput.

Change only one variable between comparison runs. Keep the seed, host, catalog,
and duration fixed when comparing scheduler configuration.

## 6. Remove the disposable database

Stop terminal A with `Ctrl-C`. From the repository root, verify the path and
then remove only this generated workspace:

```bash
test -f quickstart-work/load-test/orch8.db
rm -rf quickstart-work/load-test
```

There is no public tenant-cleanup endpoint. Never issue improvised cross-table
deletes against a shared Orch8 database.

## Checkpoint

You are done when the bounded run exits normally, final counters are printed,
metrics show activity, the engine used only the disposable database, and that
database is removed after shutdown.

Use the [load generator reference](../../../loadgen/README.md) for presets,
flags, catalog details, and longer test design.

## If something failed

| Symptom | Likely cause | Fix |
|---|---|---|
| Loadgen receives `401` | Authentication or tenant-header enforcement is still active | Use all three isolated-server overrides shown above with `--insecure` |
| Port 18080 is occupied | A prior test server is still running | Stop it or choose one port consistently for server and loadgen |
| Workload never stops | `--duration` was omitted or not forwarded | Use `npm start -- --duration=30 ...` with the separator |
| Shared data appears | Wrong server URL or database path | Stop immediately and verify both paths before rerunning |
