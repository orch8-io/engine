# Changelog

All notable changes to the Orch8 Engine will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Added

- **Trusted continuity transfer and resumable output**: execution capsules now contain a bounded, explicitly portable state model, are encrypted with tenant/continuity/epoch/destination-bound authenticated data, stored as content-addressed artifacts, signed with canonical Ed25519 manifests, and verified before import into a paused quarantine instance. The handoff API and CLI cover export, offline verification, import, accept, resume, reject, and revoke; ownership export, acceptance, and destination resume use backend transactions and monotonic compare-and-swap tokens. Capability-aware placement applies fail-closed locality and trust policy with persisted decision evidence. One-time signed continuation grants bind tenant, execution epoch, destination, action, expiry, and a replay-protected nonce. Durable stream frames use atomic contiguous sequence allocation, checkpoint linkage, reconnect cursors, explicit retractions, payload/TTL limits, and stale-epoch rejection. Effect-receipt compensation planning and provenance-chain verification are available as reusable engine services. PostgreSQL migration `058_continuity_grants_placement_streams.sql` and SQLite schema v24 add grants, placement evidence, and streams. Rust integration tests cover encrypted capsule round trips, wrong-destination rejection, single-use grants, atomic ownership/resume, stream gaps/retractions, policy routing, compensation hazards, and signature tampering; Node E2E tests cover placement, grant replay rejection, stream reconnect, and retraction.

- **Portable Continuity foundation**: introduced tenant-scoped continuity identities, monotonic execution epochs, fail-closed capsule manifests, runtime capability advertisements, compatibility previews, compare-and-swap handoff/ownership records, effect receipts, tamper-evident provenance entries, and a conservative typed-dataflow compiler. PostgreSQL migration `057_portable_continuity.sql` and SQLite schema v23 persist the protocol with tenant isolation and atomic ownership transfer. New HTTP endpoints create/query continuity executions, register bounded runtime capabilities, preview and request handoffs, resolve effect uncertainty, and inspect provenance. Handoff requests are cryptographically bound to server-generated preview evidence; stale or fabricated approvals are rejected, and public runtime registration cannot self-assert signed or attested trust. Rust storage/conformance tests and Node E2E coverage exercise ownership races, effect resolution, tenant isolation, compatibility failures, approval binding, and trust escalation.

- **Conditional Retry Policies**: `RetryPolicy` gains two optional fields for deciding whether a specific failure should actually be retried, on top of the existing `max_attempts`/backoff controls. `retry_if` is an expression evaluated against `_error.message` / `_error.details` (merged onto the instance's own `context.data` for this check only, nothing persisted) plus the instance's own `data.*` — a falsy result fails the step immediately even if attempts remain. `non_retryable_codes` is a denylist matched against `details.error_code`/`details.code` and takes priority over `retry_if`. Both fields are validated at sequence-create time (non-empty expression/codes) and covered by the existing release-diff `retry_changed` detection.

- **Template Debugger** (`POST /debug/template`): resolve a raw template string against supplied context/config/outputs fixtures without requiring a full sequence definition or stored instance. Returns the resolved value with its JSON type and per-expression provenance trace — which fallback segment supplied the value, whether a fallback fired, resolution status (ok/null/missing/redacted/error), and pipe-filter results. Complements the existing Template Resolution Inspector (which operates on stored blocks) by providing a lightweight REPL-style endpoint for template authoring.

- **Saga Block Type**: a new `Saga` composite block that runs steps sequentially with compensating actions. If a step's action fails, already-completed steps' compensations run in strict reverse (LIFO) order, best-effort. The saga node always fails after rollback — a rolled-back saga is never silently absorbed. Error context is injected before compensation begins. Supports arbitrary nesting of block types within action and compensation slots.

- **Conditional Step Guard (`when`)**: steps can now declare a `when` expression that is evaluated against the current context (`data.*`) and prior step outputs (`outputs.*`) before the step runs. If the expression evaluates to a falsy value, the step is marked `Skipped` and execution continues to the next block — no handler invocation, no output recorded. Works in both the tree evaluator and flat scheduler execution paths. Preflight checks warn on empty `when` expressions. Release diff detects `when` guard changes as `Behavioral` severity.

- **Block Output Schemas**: steps can now declare an `output_schema` (JSON Schema) that gates handler output at runtime. Conforming output proceeds normally; non-conforming output is a permanent, non-retryable failure — the output is never persisted, so downstream `{{outputs.*}}` references never see invalid data. Validation runs in all three execution paths (flat scheduler, tree evaluator, plugin dispatch). Preflight checks validate that declared output schemas are well-formed JSON Schema objects. Release diff detects output-schema changes as `Behavioral` severity. API-level validation rejects sequences whose steps carry syntactically invalid output schemas at create time.

- **Durable webhook replay protection and fuzzing**: public webhook nonces are now claimed atomically in PostgreSQL and SQLite so replay protection works across processes and restarts (Postgres migration `056_webhook_replay_nonces.sql`). New `cargo-fuzz` targets exercise expression evaluation and template resolution, and nested Cargo `target/` directories are ignored consistently.

- **Safe Workflow Release Control Plane**: promote a new sequence version with evidence instead of hope. `POST /releases` pins an exact baseline/candidate version pair and moves it through a conditional, fully audited state machine (`draft → validating → ready → canary → promoted | paused | rolled_back`), where every transition is a compare-and-swap plus an immutable decision record — concurrent promote/rollback can never both win. **Semantic diff** (`GET /releases/{id}/diff`, `POST /sequences/releases/diff`) reports what changes *operationally*: added/removed/reordered blocks, handler/queue/retry/timeout/deadline changes, router and A/B weight changes, dangling `outputs.*` references and input-schema narrowing (flagged **incompatible**), and side-effect risk. **Historical validation** (`POST /releases/{id}/validate`) replays the candidate offline against real recorded runs — no side effects, virtual time — and reports path/state divergences. **Canary routing** assigns a deterministic, monotone cohort of new instances to the candidate (same cohort key → same variant; raising the percentage never reshuffles); explicit version selection stays authoritative. **Gates** (error-rate/cancel-rate regression vs. baseline with minimum sample sizes) treat insufficient data as *inconclusive, never pass*; a failing gate during canary triggers an idempotent automatic rollback that instantly returns all new traffic to the baseline. `orch8 release create/diff/validate/canary/evaluate/promote/pause/rollback/decisions`. (Postgres migration `054_workflow_releases.sql`; SQLite schema v21.)

- **Workflow Contracts & Scenario Testing**: sequences get a first-class, version-controlled test artifact (`checkout.contracts.json` beside `checkout.json`). A contract declares input fixtures, per-handler/per-block mock policies (fixed success, fixed failure, recorded outputs, scripted attempt sequences for retry tests), signal fixtures, and expectations: terminal state, traversed/skipped blocks (ordered or not), JSON assertions over outputs and context, and call-count invariants ("charge is called at most once"). The runner executes cases on an ephemeral embedded engine where **real handlers are never invoked** and **time is virtual** — a workflow with a three-day delay tests in milliseconds. `orch8 test run` emits human, JSON, or JUnit output with CI exit codes; `orch8 test record <instance>` drafts a fixture from a real run with secrets redacted. Release validation reuses the same runner.

- **Sequence Preflight & Readiness Check** (`POST /sequences/preflight`, `GET /sequences/{id}/preflight`, `orch8 sequence preflight`): one report answering "will this sequence actually run here?" before publishing or starting it. Checks definition validity, lint, input-schema compilation, handler coverage (built-ins, enabled plugins, live workers and version pins), disabled plugins, referenced credentials (missing/disabled/expired), queue consumers (polling workers, push dispatch, routing redirects), and sub-sequence existence/publication — each reported as `pass`, `warning`, `fail`, or **`unknown`** (uncertainty is never turned into success). Findings carry evidence and remediation commands; the CLI exits non-zero for CI.

- **Stuck Instance Doctor** (`GET /instances/{id}/diagnosis`, `orch8 instance diagnose`): a ranked, read-only answer to "why is this workflow not moving?". Diagnostic codes include `WAITING_UNTIL`, `WAITING_EVENT` (with the exact missing event names and correlation key), `PENDING_APPROVAL`, `BUDGET_PAUSED`, `NO_COMPATIBLE_WORKER`, `WORKER_BELOW_VERSION_PIN`, `WORKER_NOT_CLAIMING`, `STALE_WORKER_CLAIM`, `OPEN_CIRCUIT_BREAKER`, `WAITING_CHILD`, `CHILDREN_DONE_PARENT_WAITING`, `SIGNALS_NOT_CONSUMED`, `STALE_RUNNING_STATE`, `SCHEDULER_LAG`, and `SEQUENCE_MISSING`. Diagnoses are ordered by direct evidence → probable cause → health warning with honest confidence levels; evidence that could not be collected is reported explicitly (`EVIDENCE_INCOMPLETE`) rather than ignored. Recovery actions are described, never executed.

- **Template Resolution Inspector** (`POST /sequences/inspect-template`, `GET /instances/{id}/blocks/{block}/resolved-input`, `orch8 inspect template`): shows the exact value, JSON type, and provenance every block parameter resolves to — which fallback-chain segment supplied the value, whether a fallback fired, and whether a non-string was coerced into an interpolated string — distinguishing **missing** paths from **explicit null**, **redacted** values, and **evaluation errors**. Supports a draft definition plus supplied context, or a stored instance snapshot with an `?at_block` historical boundary. Never invokes the handler; secrets never leave in clear text.

- **Webhook Delivery Inspector**: outbound webhook deliveries now record their whole story, not just the final parked row — every attempt with timestamp, HTTP status, duration, signature flag, a **normalized error class** (`dns`, `connect`, `tls`, `timeout`, `redirect_rejected`, `http_status`, ...) and a bounded, redacted error excerpt. `GET /webhooks/deliveries` (filterable by endpoint, event type, outcome, error class), `GET /webhooks/deliveries/{id}` for the attempt timeline, and `GET /webhooks/outbox/{id}/redeliver-preview` shows exactly what a redelivery would send — including whether it will be signed — without sending it. Parked outbox entries link to their attempt history; redelivery opens a new attempt group. Attempt history is GC'd after 30 days. (Postgres migration `053_webhook_delivery_attempts.sql`; SQLite schema v20.)

- **DLQ Root-Cause Fingerprinting & Guided Recovery** (`GET /instances/dlq/groups`, `orch8 instance dlq --groups`): two hundred failures with one cause now appear as **one incident group**. Failed instances are grouped by a stable, explainable fingerprint built from structured inputs (sequence identity + version, block, handler, error class/code, external status, involved resource) — never from volatile ids, timestamps, or secrets; unrecognized errors fall back to a bounded normalized-message hash. Groups show affected blocks/handlers/sequences, first/last occurrence, counts, and sample instances. **Guided recovery**: `POST /instances/dlq/groups/{fp}/retry` retries one representative sample first; bulk retry unlocks only after the sample *completes* (or an explicit `force`), is bounded, idempotent, and recomputes group membership server-side.

- **Durable Event Correlation & Joins**: the new built-in `wait_for_event` block lets long-lived workflows wait declaratively for external facts. Events are ingested via `POST /events` (or `Engine::ingest_event` when embedding) into a tenant-scoped inbox with **producer-id deduplication** — re-sending the same event is a safe no-op. Waits are keyed by `(tenant, event names, correlation key)` with `any` / `all` / `count` join policies, and resume the workflow whether the events arrived **before or after** it reached the block; matched payloads become the block's output (`{{ outputs.<block>.events.<name>.payload }}`). Consumption is one-consumer, at-most-once even under concurrent matchers; unmatched events expire after 30 days; timeout behavior reuses the `wait_for_input` timeout/escalation machinery. The definition-of-done flow — wait for `payment_received` **and** `inventory_reserved` on the same order key, in either order, rejecting duplicates — is covered by full-engine tests. (Postgres migration `055_event_correlation.sql`; SQLite schema v22.)

- **Visual Execution Workbench (API)**: `GET /instances/{id}/workbench` joins tree, unified event timeline (state transitions, block outputs, logs, audit) with a **stable ordering contract** for equal timestamps, redacted context, and payload-free output summaries into one bounded response with shareable id-only deep links. `GET /instances/{a}/compare/{b}` diffs two runs' paths and outputs; `GET /instances/{id}/fork-preview` shows — without mutating anything — which blocks a fork would copy vs. re-execute and flags the ones with external side effects. Forks now carry an explicit `sandbox` marker (dry-run by default) so sandboxes are visually separable from production runs.

- **Signed Workflow & Connector Packages** (`orch8 package keygen/build/verify/inspect/install`): a package bundles sequences, workflow contracts, and docs with a manifest declaring exactly what it needs (handlers, credentials, plugins, queues). Archives are **deterministic** (canonical JSON, sorted files) so identical content always produces identical bytes, hash, and signature; integrity is a SHA-256 content hash and authenticity an Ed25519 signature. Installs verify integrity and signature, apply a trust policy (explicitly trusted keys, or an explicit untrusted opt-in), run the packaged contracts offline, and only then upload — **under the package's own `pkg.<publisher>.<name>` namespace, never overwriting local sequences** — followed by a per-sequence preflight report showing what still needs configuring. `orch8-packages.lock` records provenance and blocks downgrades. Packages never run install-time code.

- **Shared evidence & failure model**: preflight, diagnosis, release validation, delivery inspection, and DLQ triage all report through one vocabulary — `Finding { code, severity, summary, evidence[], affected_resource, remediation[], confidence, observed_at }` — where findings are read-only and remediations are separately authorized actions. Permanent failures gain a structured `FailureEnvelope` (stable machine codes and classes alongside the preserved human message), and a reusable platform **redaction policy** (sensitive key names, secret-shaped values, headers, URL query parameters, bounded safe excerpts) guards every new operator surface.

### Fixed

- **Cross-crate failure propagation**: historical release validation, DLQ grouping, and `{{ state.* }}` template resolution now surface backend failures instead of treating missing evidence/state as an empty successful result. Mobile dedup lookup failures no longer fall through to creating a duplicate workflow, and mobile sync aborts a cycle when its local outbox cannot be read rather than sending a partial request.
- **Shutdown and cleanup observability**: server, facade-engine, and mobile lifecycle shutdown/cleanup paths now report failed background tasks and persisted dedup cleanup errors instead of silently discarding them.
- **Workbench evidence fails closed**: execution views, run comparisons, and fork previews now propagate storage failures instead of silently converting them into empty trees, outputs, logs, or audit trails that could produce false operator conclusions.
- **Storage capabilities fail closed**: required circuit-breaker, rollback, telemetry, key-value, garbage-collection, and mobile-sync operations no longer inherit silent no-op defaults. Every backend and decorator must implement or delegate them explicitly, preventing successful-looking data loss when a new storage wrapper is introduced.
- **Atomic package metadata writes**: CLI package and lock files are now written through temporary files and atomically renamed, preventing truncated metadata after interruption or disk-write failure.
- **Pool resource delete: cross-pool isolation restored**: making `DELETE /pools/{pool}/resources/{id}` idempotent had dropped the pool-membership check, allowing a resource to be deleted through another pool's URL. Deletion is now both idempotent (re-deleting a genuinely gone id returns 204) and pool-scoped (an id owned by a different pool returns 404 and is left untouched).

### Changed

- **Contemporary Rust dependency and crate hygiene**: upgraded to Reqwest 0.13 and Tonic/Prost 0.14, centralized the supported Rust version across all crates, removed unused dependencies, narrowed the `orch8` facade's default feature surface, and pinned the audited upstream `object_store` fix for the affected `quick-xml` versions until its next release.
- **Async gRPC authentication**: replaced blocking authentication inside the async request path with a Tower layer that resolves root and tenant API keys asynchronously.

### Performance

- **Concurrent mobile outbox reads**: independent status, approval, delegation, and acknowledgement batches are loaded together for each sync cycle.
- **Concurrent workbench reads**: independent execution-tree, output, log, audit, and sequence reads now run concurrently; run comparisons load both sides concurrently and use set membership instead of quadratic path scans.
- **Bounded, deduplicated hot paths**: public webhooks now enforce a bounded per-peer/slug request rate; FCM access-token refreshes use single-flight caching; and mobile HTTP clients stream responses through hard size limits instead of buffering unbounded bodies.

### Security

- **BREAKING — signed public webhooks**: `POST /webhooks/{slug}` now requires `x-orch8-signature: v1=<base64url HMAC-SHA256>` over the exact `timestamp.nonce.body` bytes. The former plaintext `x-trigger-secret` header is no longer accepted. Signatures bind the timestamp, nonce, and payload; timestamps have a bounded skew, nonces have strict syntax and durable single-use enforcement, and request bodies remain capped at 1 MiB.
- **Connect-time SSRF enforcement for mobile clients**: DNS answers are validated when establishing connections, redirects are disabled, and response bodies are streamed with strict caps to close DNS-rebinding and memory-exhaustion gaps.
- **Secret handling hardening**: push and publisher credentials use redacted secret wrappers, zeroize sensitive cached material where supported, and avoid leaking secrets through debug formatting. Push payload truncation is also UTF-8 safe.


## [0.6.0] — 2026-06-21

### Added

- **`orch8 dev` — Local Workflow Studio**: the dev command is now a full local development environment with five new capabilities: **`--server`** boots a complete HTTP API server backed by file-persistent SQLite (`.orch8/dev.db`) alongside the dev loop — the full REST API, Swagger UI, and health endpoints are available at `http://localhost:8080` (configurable via `--port`), running in insecure mode with permissive CORS for frictionless local development. **Embedded dashboard** serves the pre-built dashboard SPA from the binary itself — any path not matched by the API falls through to the SPA router, so navigating to `http://localhost:8080/` opens the dashboard with no separate process. **`--workflows <dir>`** watches a directory of `*.json` sequence files, loading all of them on startup and hot-reloading on change (mtime+size poll, same mechanism as the primary sequence file); this replaces the single-file workflow with a multi-sequence project layout. **`--auto-run`** automatically starts a new instance after each hot-reload, whether from the primary sequence file or from the workflows directory — useful for iterating on a sequence without manually re-triggering. The startup banner now shows a compact feature line (e.g. `server:8080, timers:virtual, workflows, auto-run`) so you can see at a glance what's active.

- **Step-scoped logs**: every step now has its own log stream, surfaced in the dashboard's per-execution **Logs** view (`GET /instances/{id}/logs`). Two sources feed it: **in-process** handler logs are captured by a `tracing` layer scoped to the `orch8.step` span — every event emitted while a handler runs is buffered and persisted to `step_logs` when the step's span closes, with **zero handler-side instrumentation**; and **external workers** attach `logs: [{ts, level, message}]` when completing or failing a task (`POST /workers/tasks/{id}/complete` and `.../fail`). Both land in the same store keyed by `(instance_id, block_id)`, so you can read exactly what a step logged whether it ran in-process or on a remote worker — without grepping aggregated server logs. (Postgres migration `052_step_logs.sql`; SQLite schema v18.)

- **`orch8 test replay` — recorded-run replay-diff**: replays a recorded run against a different sequence version under **virtual time** and diffs which blocks execute. The recorded run's per-block outputs are fed back verbatim (each step's handler is mocked to return what it returned originally), so the *new* version's control flow runs against the *old* run's data on an ephemeral in-process engine — no real handlers, no side effects, safe to run anywhere. The diff reports blocks the new version reaches that the recorded run didn't (`+ added`) and blocks the recorded run had that the new version skips (`- removed`), making the blast radius of a sequence refactor visible before you deploy it.

- **GitOps `orch8 sequence apply`**: a new CLI subcommand that diffs a local sequence JSON (or a directory of them) against the server and, on change, POSTs it with the **version bumped** — `unchanged` when the content matches, new sequence applied at v1, otherwise `server.version + 1`. The diff is content-only (blocks/interceptors/input_schema/sla/on_failure/on_cancel) so server-assigned identity (`id`/`version`/`created_at`) never causes a spurious apply, making it **idempotent** and safe to re-run. `--dry-run` shows what would change. A companion GitHub Action template (`docs/examples/github-action-sequence-apply.yml`) runs it on push to a `sequences/` dir, giving sequences a git-as-source-of-truth deploy flow.

- **Push-mode task dispatch**: a queue can be configured for `push` instead of the default `poll` — when a task is enqueued to it, the engine POSTs a **signed task envelope** (HMAC, the same scheme as outbound webhooks) to the queue's target URL, so workers receive work without polling. The durable `worker_tasks` row is still written, so completion is reported the usual way and a push failure just leaves the task pending (an operator can flip the queue back to `poll`). Configured per `(tenant, queue)` via `POST /queues/dispatch` (`mode`, `push_url`, optional `secret`), with `GET`/`DELETE` to manage; secrets are never echoed back. New `orch8_tasks_pushed_total` / `orch8_tasks_push_failed_total` metrics. Applied on both dispatch paths. (Postgres migration `051_queue_dispatch.sql`; SQLite schema v17.)

- **Worker version pinning**: a `(tenant, handler)` **min-version pin** — set via `POST /workers/version-pins` (`GET`/`DELETE` to manage) — makes the engine give **no tasks for that handler** to a worker reporting a version below the pin (workers already self-report `version` on poll). Lets an operator roll a fixed worker build out and stop older workers from picking up affected work, without touching the sequence. Versions compare numerically (`1.10.0 >= 1.9.0`, not a string compare; `"2"` == `"2.0.0"`) with a lexical fallback for non-numeric schemes; a worker reporting no version never satisfies a pin. Enforced on both the default and queue-scoped poll paths; lookup is best-effort (never blocks a poll on error). (Postgres migration `050_worker_version_pins.sql`; SQLite schema v16.)

- **Worker control channel**: an operator can queue control commands — `drain` (stop claiming new tasks, finish in-flight), `reload` (re-read config / re-register handlers), or `ping` (liveness probe) — for a specific worker. `POST /workers/commands` enqueues, `GET /workers/{worker_id}/commands` is the channel a worker polls (oldest-first), and `DELETE /workers/commands/{id}` acks a delivered command. Delivered via a dedicated endpoint rather than piggybacking the task-poll response, so the existing worker task-poll contract (and the SDK) is unchanged. The dashboard Workers page gains per-worker drain/reload/ping buttons. (Postgres migration `049_worker_commands.sql`; SQLite schema v15.)

- **Dynamic task-queue routing rules**: `POST/GET/DELETE /routing-rules` manage per-`(tenant, handler)` rules that override the queue an external-worker task lands on at **enqueue time** — so an operator can move a handler's traffic onto a dedicated worker queue without redeploying the sequence. Each rule has an optional `match_queue` (apply only when the task's declared queue equals this — i.e. remap queue X→Y), a `priority` (highest matching rule wins), and an `enabled` flag. Evaluated in both dispatch paths (fast-path and tree). Lookup is best-effort — a storage error falls back to the step's declared queue and never blocks dispatch. (Postgres migration `048_queue_routing_rules.sql`; SQLite schema v14.)

- **Loop/ForEach output compaction**: `retain_iterations: N` on a `Loop` or `ForEach` block bounds storage growth on long-running loops. A loop appends one output row per body step per iteration; without a bound a loop that runs for hours accumulates unbounded rows. When set, at each iteration boundary the engine keeps only the **N most recent** output rows per body block id and deletes the rest (recursing into nested composites), counting deletions in `orch8_loop_outputs_compacted_total`. Unset (the default) retains everything — fully backward compatible. Compaction is best-effort: a failure logs and the loop continues.

- **Webhook outbox**: an outbound webhook that exhausts all its retries is now **parked** in a `webhook_outbox` table instead of being silently dropped (the previous behavior — `orch8_webhooks_failed_total` ticked and the event was lost). `GET /webhooks/outbox` lists parked deliveries (url, event type, attempts, last error); `POST /webhooks/outbox/{id}/redeliver` runs a fresh signed send-with-retry pass and removes the row on success (or returns **502** and keeps it parked on failure); `DELETE /webhooks/outbox/{id}` discards one. A new `orch8_webhooks_parked_total` metric counts parkings. The engine wires the outbox sink once at scheduler startup (`init_outbox`), so parking is transparent to every emit site. The dashboard Operations page gains a "Webhook outbox" section with redeliver/discard. (Postgres migration `047_webhook_outbox.sql`; SQLite schema v13.)

- **Parent/child execution navigation**: `GET /instances/{id}/children` lists the sub-sequence instances a given instance spawned (those whose `parent_instance_id` matches). `parent_instance_id` was already on every instance response; the dashboard InstanceDetail now renders an "↑ parent" breadcrumb link when the instance is a child, and a "Child executions" table linking to each sub-execution — so a fan-out of sub-sequences is navigable in both directions instead of being a flat, disconnected list.

- **Batch actions by filter**: `POST /instances/batch-action` applies one control action — `retry`, `pause`, `resume`, `cancel`, or `signal` — to every instance matching a filter (`tenant_id` required; supports `namespace`, `sequence_id`, `states`, and the indexed `metadata` equality map). `dry_run: true` returns the match count without applying; `limit` caps the blast radius (default 1000, max 10000). The response reports `matched`/`applied`/`skipped`/`failed`, and every applied action writes a `batch_action` audit-log entry. `retry` re-runs `Failed` instances (clears the stale tree + sentinel outputs); the rest enqueue the matching signal atomically (terminal targets are skipped, not failed). The dashboard Executions page gains a batch-action bar over the current filter with a match-count preview before applying.

- **Sequence cleanup hooks (`on_failure` / `on_cancel`)**: a sequence may declare best-effort cleanup block lists that run when an instance reaches terminal `Failed` / `Cancelled`. Each top-level step block is dispatched once with the instance's final context; **errors are swallowed** (the instance is already terminal — cleanup must never resurrect or wedge it). `on_failure` fires from every failure path (fast-path step failure and tree-evaluation failure); `on_cancel` fires from both signal-driven cancellation and evaluation-completed cancellation. Answers the HN complaint that when a workflow dies "the world just seems to die" with no chance to release a lock or send a notification. (v1 runs step blocks only; non-step blocks are skipped with a warning. Postgres folds the lists into the `definition` JSON blob, no migration; SQLite adds `on_failure`/`on_cancel` columns, schema v12.)

- **Stuck-instance / SLA breach alerts**: a sequence may declare an alert-only `sla` policy — `max_runtime` (wall-clock lifetime from `created_at`) and/or `max_step_runtime` (current step's runtime). Every tick, a bounded sweep over active instances (`Scheduled`/`Running`/`Waiting`/`Paused`) emits an `instance.sla_breached` webhook and increments `orch8_sla_breached_total{type=max_runtime|max_step_runtime}` when a bound is exceeded — **once per breach**, de-duplicated by a reserved `_sla:*` sentinel block output. The instance is **never** failed or paused (alert-only; contrast `deadline` on a step, which fails it). InstanceDetail shows an "SLA breached" badge + a breach panel; the sweep skips all per-instance work when no instances are active. Answers the HN complaint that a stuck workflow "just seems to die" with no signal. (Postgres: `sla` folded into the sequence `definition` JSON blob, no migration. SQLite: `sla` column, schema v11.)

- **Indexed metadata search**: `GET /instances?metadata.<key>=<value>` filters the instance list by top-level `metadata` keys — repeatable params AND together (`?metadata.env=prod&metadata.team=core`). On Postgres this is a JSONB containment (`@>`) query served by the **already-present GIN index** on `metadata` (`idx_instances_metadata`, `jsonb_path_ops`) — no external search index needed. On SQLite it falls back to `json_extract` text-equality with the key bound as a quoted JSON-path member (literal top-level match, mirroring Postgres containment). String-equality semantics. The dashboard Executions page gains a metadata key/value filter pair. No migration needed — the index has existed since `009_create_indexes.sql`.

- **Cron next-fires preview + schedule UI**: new `GET /cron/{id}/next-fires?n=5` (max 50) returns the upcoming fire instants (UTC) for a schedule, computed by iterating the DST-correct `calculate_next_fire_after` — each instant is fed back as the next "after" bound, so gap occurrences clamp and fall-back ambiguity fires once, exactly as the engine will. The dashboard Cron page gains an expandable row per schedule showing the next 5 fires rendered in the schedule's own timezone (with the UTC instant alongside), plus the schedule's `overlap_policy` and a skipped-fires count/last-skip badge. Pause/resume was already wired (the `enabled` toggle). Lets operators verify a schedule before trusting it.

- **Dashboard: Run a sequence**: the Sequence detail page now has a **Run** button (active versions only) that opens an inline `context.data` JSON editor and starts a new instance via `POST /instances`, then navigates straight to the new Instance detail page. When the sequence declares an `input_schema`, the editor is pre-shaped from the schema's property defaults (or type-appropriate empties), and a payload that fails validation surfaces the engine's 422 inline. Closes the "no way to start a run from the UI" gap. Pairs with `input_schema`.

- **Sequence `input_schema` + create-time validation**: a sequence may declare an optional `input_schema` (a JSON Schema object). When present, `context.data` of every new instance is validated against it at create time — both the single `POST /instances` and batch create reject a malformed payload with **422 Unprocessable Entity** (with JSON-pointer paths for each violation) before the instance is persisted, so bad input never reaches the engine. The schema itself is checked for well-formedness at sequence-create time (**400** on a broken schema), not on the first instance. A corrupt stored schema fails open (logged, validation skipped) rather than wedging the sequence. The schema also doubles as the contract the dashboard renders a Run form from.

- **Cron overlap policies + skip metric**: `overlap_policy` on `CronSchedule` controls what happens when a fire is due while a previous run from the same schedule is still active — `allow` (default, unchanged behavior), `skip` (count it: `skipped_fires` + `last_skipped_at` on the schedule, `orch8_cron_skipped_total` metric), `buffer_one` (defer until the previous run finishes, then fire once), `cancel_previous` (cancel still-active runs, then fire). Previous runs are attributed via a `metadata.cron_schedule_id` stamp the cron loop now writes on every fire. (migration 046, SQLite schema v9)

- **Environment banner**: `ORCH8_ENV_LABEL` + `ORCH8_ENV_COLOR` render a colored banner strip across the dashboard (e.g. a red `PRODUCTION` ribbon), served by the new unauthenticated `GET /info` endpoint alongside the engine version.

- **DST-correct cron schedules**: explicit, unit-tested semantics for both DST edges. Nonexistent local times (spring forward — a 02:30 schedule on the day 02:00 jumps to 03:00) fire at the **first valid instant after the gap** instead of silently skipping the day; ambiguous local times (fall back) fire **once**, at the first occurrence. Gap detection re-evaluates the expression against a fixed offset frozen at "now" and clamps occurrences that fall inside the gap. Tested against both hemispheres (`America/New_York`, `America/Santiago`) under virtual time.

- **Signal injection on resume-from and fork**: both `POST /instances/{id}/resume-from/{block_id}` and `POST /instances/{id}/fork` accept an optional `signals: [{signal_type, payload}]` array, enqueued before the instance becomes claimable — resume-from enqueues before the wake transition, fork now creates the clone un-claimable (`next_fire_at` unset), seeds outputs and signals, then arms it. Performs the `initialize` → `notifications/initialized` → request handshake per call, supports `tools/call` and `tools/list`, parses both `application/json` and `text/event-stream` (SSE) responses, and honors `Mcp-Session-Id`. Because it runs as a durable step, MCP tool calls inherit retry, rate-limiting, circuit-breaker, and crash-recovery semantics — a tool reporting `isError: true` surfaces as `is_error` rather than failing the step.

Engine-local and offline — the managed multi-tenant catalog remains a cloud concern.

- **Durable artifacts**: a generic binary-blob primitive (`ArtifactRef` + `ResourceStore::{put,get,delete,list}_artifact`) backed by [`object_store`]. Bytes live out-of-band; a small JSON ref travels in step outputs (survives checkpoint/replay). Backends: **local filesystem**, **S3-compatible** (AWS S3 / Cloudflare R2 / MinIO), configured via `ORCH8_ARTIFACT_BACKEND` / `[artifacts]`. In-memory is intentionally not a production backend (it would break durability). Artifact bytes are **encrypted at rest (AES-256-GCM)** by the `EncryptingStorage` wrapper before reaching the object store — independent of bucket SSE. Encrypted blobs carry a magic header so reads are self-describing: any blob written *before* encryption was enabled is returned as-is rather than failing to decrypt, and `ArtifactRef.size` reports the plaintext length. On ephemeral container filesystems (Cloud Run, k8s without a PVC) artifacts are lost on restart — the server now logs a startup `WARN` and **requires an absolute `artifacts.path`**. Use the S3 backend for cloud deployments.
  - **Automatic retention sweeper**: set `ORCH8_ARTIFACT_RETENTION_SECS` (`engine.artifact_retention_secs`, default `0` = disabled) to have the background GC delete the artifacts of instances that have been in a terminal state longer than the window. Bounded per tick, idempotent (an `_artifacts_gced` instance marker prevents re-scanning), and surfaced via `orch8_gc_artifacts_{deleted,errors}_total`. Disabled by default so artifacts are never deleted unless you opt in.
- **Binary I/O on `tool_call`**: `response_as: "artifact"` captures a (2xx) response body as a durable artifact; `body_artifact` (+ optional `upload: { mode: "multipart", … }`) uploads an artifact's bytes as the request body (raw or multipart). Generic — composes into image generation, asset fetch, and media upload without any provider-specific handler.
- **Dry-run mode**: pass `dry_run: true` when creating an instance (per-run, so *any* sequence can be run dry). Side-effecting steps — `tool_call`, `llm_call`, `mcp_call`, `agent`, `emit_event`, `send_signal`, `embed`/`memory_store`/`memory_search`, `self_modify`, and the Activepieces/gRPC plugins — run their **pure validation first** (URL/SSRF safety, API-key presence, `tool_name`, trigger resolution incl. tenant + target sequence, target-instance existence, block-definition validity) and only then skip the real effect, returning **one canonical envelope**: `{ "dry_run": true, "handler": <name>, "would": <skipped effect>, …<success-shape with empty values> }` so downstream templates resolve and a green dry-run actually means the step is well-formed. Control flow, templating and branching execute for real; `human_review` skips its outbound notification and, by default, keeps its pending/pause gate. Pass `dry_run_auto_approve: true` (alongside `dry_run`) to auto-resolve human gates with the default (first) choice so the simulation flows through and exercises post-gate steps. The flag lives in the per-instance `RuntimeContext` (queryable on the instance, logged at creation, survives checkpoint/replay) and propagates to spawned sub-sequence children, so nothing leaves the box. **Dry-run is part of the idempotency identity** — a real run reusing a dry-run's `idempotency_key` is *not* deduplicated to the simulation.
  - **Rollback caveat**: `dry_run` lives in the serialized instance context; rolling the binary back to a pre-dry-run version drops the flag (serde ignores unknown fields), so in-flight dry-run instances would execute their remaining steps **for real**. Drain dry-run instances before downgrading.
- **`blob_put` / `blob_get` handlers**: first-class step access to the durable-artifact primitive (previously reachable only via `tool_call`). `blob_put` stores `text` or `data` (base64) and returns an `ArtifactRef`; `blob_get` reads it back as `base64` or `utf8`. Size-guarded (`max_size_bytes`, default 25 MiB), dry-run aware, and accepts a `ref` as a bare key string, `{ "key": … }`, or the `{ "artifact": { … } }` wrapper that `blob_put`/`tool_call` emit — so bytes pass between steps via `{{outputs.<id>.artifact}}`. Makes binary pipelines (LLM → image → store → publish) expressible without a provider-specific handler. **Requires an artifact backend** to be configured (see *Durable artifacts*); fails permanently with `Unsupported` otherwise.
- **Signed outbound webhooks**: `webhooks.secret` (a redacted `SecretString`) signs every delivery Stripe/GitHub-style — `X-Orch8-Timestamp` + `X-Orch8-Signature: sha256=HMAC-SHA256(secret, "{timestamp}.{body}")` — so receivers can verify authenticity and reject replays. Mirrors the inbound trigger-secret model in the outbound direction; deliveries stay unsigned (unchanged) when no secret is set. Builds on the existing durable retry/backoff delivery path.
- **Per-tenant API keys**: admin-only endpoints (`POST /api-keys`, `GET /api-keys`, `DELETE /api-keys/{id}`) mint, list, and revoke scoped authentication keys. Each key is bound to exactly one tenant; the plaintext secret (shown once at creation) is never stored — only its SHA-256 hash is persisted. Per-tenant keys authenticate via the same `x-api-key` header but ignore `X-Tenant-Id` (the tenant is taken from the key record), preventing header spoofing across tenants.
- **`last_used_at` tracking for API keys**: every successful authentication with a per-tenant key updates `last_used_at` via a fire-and-forget async write, giving operators an audit trail for stale-key rotation.
- **gRPC per-tenant API key parity**: the gRPC interceptor (`Orch8GrpcService`) now resolves per-tenant keys by hash against storage, matching the HTTP middleware contract. Root key remains unscoped; per-tenant keys reject mismatched `x-tenant-id` metadata with `PermissionDenied`.

### Changed

- **Instance state machine — terminal escape hatch** (#59): `Completed → Scheduled` and `Cancelled → Scheduled` are now valid transitions (required by resume-from-block). Side effect: `PATCH /instances/{id}/state` can also re-schedule completed/cancelled instances. All other exits from terminal states remain invalid; `is_terminal()` is unchanged.
- **`StorageError` taxonomy**: split the catch-all `Unsupported` into permanent `Unsupported` (not configured / not supported → maps to a non-retryable step error and HTTP 500) vs transient `Backend` (object-store network/throttle failure → retryable, HTTP 503). Fixes artifact misconfiguration retrying forever.
- `tool_call` `response_as:"artifact"` with a non-idempotent method requires an idempotent endpoint: if the artifact store fails *after* a successful request, the step retries and re-sends.
- **Secure-by-default tenant isolation**: `require_tenant_header` now defaults to `true` (was `false`). Disabling it still works for single-tenant deployments but the server logs a loud `WARN` at startup when auth is enabled.
- **Encryption fail-closed**: the server now refuses to start if no encryption key is configured, unless `--insecure` is explicitly passed. A missing key previously stored credentials and `context.data` in plaintext silently.
- **Unified step-dispatch internals**: the flat-sequence (`scheduler::step_exec`) and tree-evaluator (`handlers::step_block`) dispatch paths now share one `prepare_step` (context snapshot → templates → credentials → cache key, with its failure logging) and one `breaker_preflight` (circuit-breaker open/fallback/defer decision). The two ~150-line preambles were previously hand-mirrored and drifted; they are now single-sourced, with each path keeping only its own node-level vs instance-level failure transition.

### Fixed

- **`wait_for_input` timeout baseline reset on every wake** (deep review 2026-06): the flat-path scheduler re-stamped `current_step_started_at` on *every* entry into a step, so a Waiting instance woken by the deadline sweep reset its own timeout baseline — the timeout only ever fired through a stale-snapshot accident (the gate check read the claim-time instance, not the fresh stamp). That same staleness fired gates *prematurely*: a gate reached after a slow earlier step inherited that step's stamp and could time out on first entry. The stamp is now first-entry-only and the gate check evaluates against the durable per-step baseline. On the tree path the gate now uses the node's own `started_at` (the shared `current_step_started_at` slot is overwritten by parallel branches). Regression test: `wait_for_input_baseline_survives_reentry`.
- **Deadline sweep woke instances on the wrong step's timeout**: `process_waiting_deadlines` checked every `wait_for_input` step in the sequence against the *current* step's start time, so a sequence with two or more human gates churned through spurious wake/re-park cycles (one per tick) while waiting on a later gate. The wake check now requires `runtime.current_step` to match the step being checked.
- **Concurrency-limit enforcement scrambled claim priority**: deferring over-limit instances used `swap_remove`, which pulls tail elements into the holes — surviving instances lost the priority order `claim_due_instances` established (deferring indices [1,3] from `[A,B,C,D,E]` yielded `[A,E,C]`). Replaced with order-preserving filtering. Regression test: `enforce_concurrency_limits_preserves_claim_order`.
- **SQLite `recover_stale_instances` left recovered instances unclaimable**: unlike Postgres, the SQLite recovery UPDATE did not reset `next_fire_at`, so a wedged instance carrying a far-future fire time stayed parked until that time naturally arrived after recovery. Now resets `next_fire_at = now` (parity with Postgres). Regression test: `recover_stale_resets_next_fire_at`.
- **SQLite node `started_at` overwritten on re-dispatch**: `update_node_state`, `update_nodes_state`, and `batch_activate_nodes` lacked the `COALESCE(started_at, …)` guard the Postgres backend documents and applies, so any Waiting→Running re-dispatch (worker retry, wake) clobbered the node's original start time — corrupting per-step latency metrics and the tree-path `wait_for_input` baseline. Regression tests: `update_node_state_preserves_started_at_on_redispatch`, `batch_node_updates_preserve_started_at`.
- **SQLite read-then-write races under DEFERRED transactions**: `claim_worker_tasks_from_queue(_for_tenant)` could hand the same pending task to two concurrent pollers, and concurrent `merge_context_data` calls could silently drop each other's keys — both did SELECT-then-UPDATE inside sqlx's default `BEGIN DEFERRED`, where concurrent reads don't serialize. Both now use the `BEGIN IMMEDIATE` convention already established by `claim_due` / `workers::claim`; `create_instance_with_dedupe` was converted too (its unique constraint already protected correctness, but DEFERRED upgrades could fail with `SQLITE_BUSY_SNAPSHOT` instead of cleanly serialising). Regression tests: `concurrent_queue_claims_do_not_double_claim`, `concurrent_context_merges_lose_no_keys`.
- **SSE output stream skipped same-millisecond outputs**: the `/instances/{id}/stream` poll cursor used a strict `created_at >` bound, so when a batch of step outputs shared the cursor's exact timestamp, everything at that timestamp beyond the first poll was never delivered. The storage bound is now inclusive (`>=`) and the stream deduplicates boundary rows by output id. Regression test: `outputs_after_created_at_includes_cursor_timestamp`.
- **Mobile sync `upsert_sequence` swallowed delete failures**: a failed `delete_sequence` was discarded with `let _ =`, after which `create_sequence` either violated the unique constraint or left duplicate rows while the sync reported success. The delete error now propagates.
- **Tree-path step counter lost increments**: the tree evaluator bumped `total_steps_executed` via a non-atomic read-modify-write that also swallowed `get_instance` errors with `.ok().flatten()` — two steps completing in the same tick could lose an increment (under-counting `max_steps_per_instance` enforcement). Now uses the same atomic `increment_total_steps` as the flat path.
- **Webhook OpenAPI contract mismatch**: `POST /webhooks/{slug}` documented `201 Created` but has always returned `202 Accepted`; the spec now matches the implementation.
- **`orch8 init` scaffold was invalid against the engine wire format** (#57): the generated `sequence.json` used `"type": "Step"` (block tags are snake_case `step`) and `"delay": "5s"` (`DelaySpec` requires `{"duration": <ms>}`), so the documented first-run sequence was rejected by `POST /sequences`. The default template now produces a sequence the engine actually accepts. (Note: `docs/QUICK_START.md` still shows the old `"delay": "2s"` shape — pending docs fix.)
- **Wide-`send_window` e2e test failed nightly between 23:00–24:00 UTC**: `end_hour` is exclusive and sequence validation rejects `start_hour == end_hour` (which the scheduler would treat as a 24h window), so a full-day window is inexpressible and the test's 0–23 window excluded the last hour of the day. The test now uses a wide wrapping window centered on the current UTC hour. The validation/runtime disagreement on `start == end` is left as a known inconsistency.
- **Infinite retry on the fast path (never reached the DLQ)**: a retryable step failure in the flat-sequence scheduler deleted *all* of the block's outputs without persisting a retry marker, so `compute_attempt` reset to `0` every tick and `attempt >= max_attempts` was never true — the step retried forever and never failed the instance. The fast path now writes a `__retry__` marker (mirroring the tree evaluator) so the attempt counter advances and the instance fails after `max_attempts`. Retry markers are excluded from the completed-block set (SQLite + Postgres) so the block still re-executes, while the `__in_progress__` crash-recovery sentinel still counts as completed.
- **Response-body OOM**: `http_request` and `tool_call` checked `Content-Length` (trivially omitted by a chunked response) and then buffered the *entire* body before the post-read size check, so an endpoint streaming gigabytes could OOM the worker. Both now stream with a hard 10 MB cap that aborts mid-stream, bounding peak memory.
- **Silently stranded instances**: when no concurrency permit was available (or when re-arming a signalled instance), the scheduler dropped the result of the `Running → Scheduled` deferral write with `let _ = …`. If that write failed the row stayed `Running` and was never re-claimed, with no log. Both paths now `warn!` on failure so a stuck instance is observable.
- **Mobile sync silently no-op'd under encryption** (security review 2026-06): `EncryptingStorage`'s `MobileSyncStore` impl was empty `{}`, so it inherited the trait's no-op default methods — on any encryption-enabled server (the fail-closed production default) every mobile device/status/approval/command read returned empty and every write was discarded. The wrapper now delegates all mobile methods to the inner backend (and encrypts command payloads, see Security). Regression test: `mobile_methods_delegate_not_noop`.
- **gRPC `update_context` / `complete_task` bypassed the context-size cap** (security review 2026-06): the HTTP path enforces `max_context_bytes` via `check_size`, but the gRPC handlers only had the 10 MB raw-JSON guard — a gRPC client could grow an instance's context past the configured semantic limit, and that bloated context then rides every scheduler claim. The gRPC service now carries `max_context_bytes` (wired from `config.engine`) and validates context size on `update_context` and after the worker-output merge in `complete_task`.
- **gRPC `get_sequence_by_name` missing post-fetch tenant check** (security review 2026-06): every other `get_*` RPC and the HTTP counterpart validate the fetched row's tenant after the storage query; this one relied on the pre-filter alone. Added the defense-in-depth post-fetch check so a future storage bug can't let one transport leak a cross-tenant row the other rejects.

### Performance

- **Subtree cancellation was O(n²)** (deep review 2026-06): `cancel_subtree` scanned the entire execution tree once per visited node to find children, and then issued one UPDATE per cancelled descendant. It now builds a parent→children index (O(n) BFS) and cancels all descendants in a single batched `update_nodes_state` round-trip — large `ForEach`/nested `Parallel` cancellations drop from hundreds of queries to one.
- **SQLite fork-from output copy was N+1**: `copy_block_outputs` issued one INSERT per copied row; a fork over hundreds of completed steps meant hundreds of round-trips, with partial copies visible on crash mid-loop. Now a chunked multi-row INSERT inside one transaction. Regression test: `copy_block_outputs_copies_all_rows`.
- **Idle deadline sweep**: `process_waiting_deadlines` ran its per-instance sequence-cache walk on every tick even when no instances were Waiting; it now returns immediately on an empty batch.
- **SLA deadline check**: `check_sla_deadlines` re-flattened the whole block tree (`flatten_blocks`) on every iteration of the evaluate loop (up to 200×). It now reuses the `block_map` the evaluator already builds once per evaluation — removing an O(blocks) allocation + hash per iteration on composite instances.
- **API-key auth hot path**: per-tenant API keys are now resolved through a process-wide short-TTL cache (`moka`, keyed by SHA-256 hash, 30 s), shared by the HTTP middleware and the gRPC interceptor. This collapses a per-request indexed `SELECT` — and, on the HTTP side, an unbounded per-request `last_used_at` write — into at most one read + one write per key per window. Revocation stays immediate: `revoke_api_key` evicts the cache entry (via `RETURNING key_hash`) in the same statement that flips the flag. The root key is also hashed once at startup (precomputed digest) instead of on every request.
- **Expression engine — borrow instead of clone**: the expression interpreter now threads `Cow<Value>` so path resolution returns a *borrow* into context/outputs. A value that is only inspected (`len`/`count`, comparison, truthiness) is no longer deep-cloned — `len(outputs.big_array)` on a 200-element array dropped from **57 µs to 0.57 µs (~100×)**. `runtime.*` access also stops serializing the entire `RuntimeContext` to JSON on every read (common scalar fields resolved directly from the typed struct; **−77%** on `runtime.attempt`).
- **Scheduler per-step writes**: each completed step on the fast path did a full `get_instance` read **plus** a whole-`context` rewrite just to bump `total_steps_executed`. Replaced with an atomic `increment_total_steps` (a single targeted `json_set` / `jsonb_set` UPDATE that returns the new count and leaves the rest of `context.data` untouched), removing ~2 DB round-trips and a full-context serialization per step.
- **Templated-step state fetch**: `resolve_templates_in_params` fetched and serialized the entire instance KV map for *every* templated step, even when no template referenced `{{ state.* }}`. The fetch is now gated on an actual `state` reference (conservative — no false negatives), skipping a DB round-trip for the common outputs-/context-only case.

### Security

- **Cross-tenant sequence disclosure via `GET /sequences.json`** (security review 2026-06): the plain-array sequence-list endpoint (used by mobile SDKs) ignored the caller's tenant entirely and returned up to 1000 sequence definitions across *all* tenants — block logic, prompts, integration params. It now scopes to the caller's tenant exactly like the paginated `GET /sequences`; only a global/root key sees across tenants.
- **Trigger HMAC secrets stored in plaintext** (security review 2026-06): `TriggerDef.secret` (the key that authenticates inbound signed webhooks from Stripe/GitHub/etc.) was bound straight into the `triggers` table while credentials were encrypted — a DB dump or replica leak exposed every signing secret, letting an attacker forge valid webhook signatures. `EncryptingStorage` now encrypts the secret on write and decrypts on read (same shape as credentials). Regression test: `trigger_secret_encrypted_at_rest_decrypted_on_read`.
- **Resolved credentials persisted in plaintext in `mobile_commands`** (security review 2026-06): a mobile step-delegation's `payload` — which can contain `credentials://` references resolved to real API keys/tokens — was stored unencrypted and survived until ack. `EncryptingStorage::create_mobile_command` now encrypts the payload at rest; `fetch_pending_commands` decrypts it so the device still receives plaintext over TLS. Regression test: `mobile_command_payload_encrypted_at_rest`.
- **Cross-tenant mobile command injection** (deep review 2026-06): `POST /mobile/commands` performed no tenant check on the request's `device_id`, so any tenant-scoped API key could enqueue arbitrary commands — including `complete_step` — into another tenant's device queue. The handler now verifies the caller's tenant owns the target device (404 otherwise), with the same root-key/insecure-mode fall-open convention as `resolve_approval`.
- **Workflow DoS via deep nesting, huge arrays, and unbounded mobile sync** (security review 2026-06): sequence validation now caps block-tree nesting at depth 64 (`MAX_NESTING_DEPTH`) so a maliciously deep tree is a clean validation error instead of a stack overflow on the worker thread; the `unique()` expression built-in was rewritten from O(n²) to O(n) and `unique`/`sort` refuse arrays over 100k elements; and `POST /mobile/sync` caps each request array (status updates, approvals, delegations, acks) at 500 items so one device can't serialize the DB with a single request. Regression tests: `validation_rejects_pathological_nesting`, `array_fns_refuse_oversized_input`.
- **SSRF gaps on secondary HTTP clients** (security review 2026-06): the outbound webhook-delivery client and the Activepieces sidecar client followed redirects with reqwest's default policy (no hop re-validation), so a redirect to `169.254.169.254`/internal hosts was followed even though the primary handler clients re-check every hop. Both now apply the same redirect-re-validation policy. Separately, the `grpc_plugin` and `human_review` handlers ran their SSRF check *after* the dry-run skip, so a dry-run gave a false green on a blocked URL — the check now runs before the skip (validate-then-skip), matching every other handler.
- **Outbound header injection** (security review 2026-06): the `http_request`, `tool_call`, `mcp_call`, and `human_review` handlers passed workflow-supplied header *names* straight to reqwest. They now reject `Host` (overriding it can reroute a request behind a shared proxy) and any name containing control characters (request-splitting), via a shared `outbound_header_name_allowed` guard.
- **SSRF via HTTP redirects**: the shared outbound HTTP client (`http_request`, `tool_call`, `agent`, `mcp_call`) followed redirects with reqwest's default policy, re-validating nothing — a public URL could `302 → http://169.254.169.254/…` and reach cloud-metadata/internal hosts. The client now re-checks every redirect hop and refuses redirects to non-`http(s)` schemes or private/loopback/link-local IP literals (hop count also capped).
- **`ORCH8_ALLOW_INTERNAL_URLS` fail-open**: the SSRF-bypass flag was enabled by `is_ok()` — true for any value, including an empty / `0` / `false` leftover var. It now requires an explicit truthy value (`1`/`true`).
- **Artifact IDOR in `tool_call`**: a `body_artifact.key` was fetched verbatim, so a workflow could read another instance's (potentially another tenant's) artifact via a sibling key. The key must now be prefixed by the current instance's id.
- **Artifact IDOR in `blob_get`**: `blob_get` now enforces the same instance-prefix ownership check as `tool_call`, rejecting cross-instance artifact reads before touching storage.
- **`api_key_env` secret exfiltration guard**: `llm_call` and `memory` handlers block workflow-controlled `api_key_env` from reading the engine's own secrets (`ORCH8_*` namespace) and well-known infrastructure env vars (`DATABASE_URL`, `AWS_SECRET_ACCESS_KEY`, `GITHUB_TOKEN`, etc.). Only legitimate provider keys (e.g. `OPENAI_API_KEY`) are allowed. The guard now also rejects any env var whose name *contains* a secret-shaped substring (`SECRET`, `PASSWORD`, `CREDENTIAL`, `PRIVATE_KEY`, `SESSION_TOKEN`), catching the long tail of third-party host secrets (`STRIPE_SECRET_KEY`, `TWILIO_AUTH_TOKEN`, …) an exact-match denylist can't enumerate.
- **Expression engine hardening**: `random()` with extreme `i64` bounds (`i64::MIN .. i64::MAX`) no longer panics from overflow — rewritten with `i128` arithmetic. `slice(arr, start, end)` with `start > end` now returns an empty array instead of panicking on an invalid Rust range.
- **Webhook future-timestamp window**: inbound webhook replay protection accepted timestamps up to 5 minutes in the *future* (symmetric `abs_diff`). Future skew is now capped at 60 s while the 5-minute past window is retained, shrinking the window for pre-dated captured payloads.
