## 2024-11-20 - Removed synchronous eprintln! from async executor
**Learning:** Found leftover debugging `eprintln!` statements inside core engine loops in `signals.rs` and `handlers/builtin.rs`. In a high-throughput async executor like Tokio, synchronous console I/O can block executor threads and degrade performance.
**Action:** Replaced `eprintln!` with `tracing::debug!` which integrates cleanly with the existing tracing subscriber, honors logging levels in production, and avoids unexpected synchronous stalls in async functions.
## 2026-04-19 - Avoided unnecessary cloning when building sqlx QueryBuilder
**Learning:** In `orch8-storage/src/postgres`, multiple functions (like `apply_instance_filter`, `apply_worker_task_filter`) were cloning strings (`tid.0.clone()`, `ns.0.clone()`, etc.) when pushing bind parameters to `sqlx::QueryBuilder`. Since `QueryBuilder::push_bind` accepts references and internally binds the lifetime, the explicit `.clone()` is unnecessary and creates overhead.
**Action:** Replaced `.push_bind(var.clone())` with `.push_bind(var)` (or `.push_bind(&var)` for inner properties) and updated the function signatures to include a named lifetime parameter `<'a>` to link the `QueryBuilder` and the filter reference.
## 2026-04-19 - Avoided unnecessary cloning when building sqlx QueryBuilder in SQLite backend
**Learning:** Just like the PostgreSQL backend, the SQLite backend in `orch8-storage/src/sqlite` was doing unnecessary `.clone()` calls when binding strings to `sqlx::QueryBuilder` instances in `apply_worker_task_filter` and similar helper methods. Since `QueryBuilder::push_bind` accepts references (`&str`), cloning strings before passing them is pure overhead and can be avoided by passing references and correctly linking the lifetimes.
**Action:** Replaced `.push_bind(var.clone())` with `.push_bind(&var)` across SQLite storage queries, and updated signatures to use named lifetimes (e.g. `<'a>`) to tie the `QueryBuilder` to the data reference.
## 2024-11-20 - Eliminated DashMap write-lock contention on circuit breaker hot paths
**Learning:** The circuit breaker state in `orch8-engine` is heavily accessed on every step execution (check, then record success). Previously, `check` used `entry(key).or_insert_with(...)` and `record_success` used `get_mut(&key)`, which acquire write locks on the underlying `DashMap` shards even when the circuit is completely healthy and requires no mutation. This serialized concurrent execution streams mapping to the same shard.
**Action:** Implemented a fast-path read-only check (`self.breakers.get(&key)`) for the happy path (breaker `Closed` with no failures). If the condition holds, the function returns immediately without ever requesting a write lock, allowing thousands of concurrent healthy checks to proceed in parallel without contention.
## 2024-11-20 - Avoided string allocations when filtering lists into temporary HashMaps
**Learning:** In `orch8-engine/src/scheduler.rs`, inside `enforce_concurrency_limits`, a temporary `HashMap<String, Vec<usize>>` was used to group instances by `concurrency_key`. This forced string cloning (`key.clone()`) on every item processed, which is pure overhead on a hot path, since the temporary map is discarded before the function returns.
**Action:** Replaced `HashMap<String, Vec<usize>>` with `HashMap<&str, Vec<usize>>` by using `key.as_str()` during population. This uses borrowed keys pointing back to the incoming batch data, eliminating heap allocations for strings entirely while building the lookup table.
## 2024-11-20 - Avoided string allocations when filtering instances by concurrency key in database layers
**Learning:** In both PostgreSQL and SQLite storage layers (`orch8-storage/src/postgres/instances.rs` and `orch8-storage/src/sqlite/instances.rs`), when grouping candidates by `concurrency_key` inside `filter_by_concurrency`, the code was cloning the concurrency key string to insert it into a temporary `HashMap<String, Vec<usize>>`. Since the original `candidates` slice lives for the entire duration of the function, the cloned string is pure overhead.
**Action:** Replaced `HashMap<String, Vec<usize>>` with `HashMap<&str, Vec<usize>>` by using `key.as_str()` during insertion. This borrows the string reference instead of allocating on the heap, ensuring memory efficiency during this hot database concurrency grouping phase.
